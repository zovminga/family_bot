# SPEC — дашборд семейных трат

Контекст и спецификация: что собрано, как считается, какие приняты решения.
Цель файла — чтобы спустя время можно было быстро восстановить логику.

Дата составления: 2026-06-27. Язык доков — русский, комментарии в коде — английский
(по конвенциям проекта).

---

## 1. Назначение

Автономный HTML-дашборд по семейным тратам. Источник данных — та же Google-таблица
`FamilyExpenses` (лист `Data`), куда пишет Telegram-бот `family_bot`, либо её
CSV-экспорт. На выходе — один самодостаточный `.html` со вшитыми данными
(открывается двойным кликом, можно переслать одним файлом).

Главный акцент дашборда (после итераций с пользователем):
**не «сколько всего потрачено», а «какие статьи идут по месяцам» и
«что можно/нельзя оптимизировать»**. Тоталы вторичны.

## 3. Пайплайн данных (`build_dashboard.py`)

Поток: **источник → парсинг → доначисления → курсы → нормализация → фильтр
периода → группировка тем → вшивание JSON в шаблон**.

### 3.1 Источник
- По умолчанию — Google Sheets (`gspread` + service account, лист `Data`).
- Альтернатива — CSV-экспорт через `--csv "<path>"` (так и собираем сейчас,
  т.к. локально нет `creds.json`).

### 3.2 Заголовки
Колонки матчатся по алиасам (RU/EN, регистронезависимо) — `HEADER_ALIASES`:
`date/дата`, `month/месяц`, `category/категория`, `amount/сумма`,
`currency/валюта`, `who/кто внес/spender`, `comment/комментарий`.
Обязательные: `amount`, `currency`, `category` (иначе ошибка).

### 3.3 Парсинг сумм (`parse_amount`)
Терпит пробелы, неразрывные пробелы, запятую как десятичный разделитель.
Если есть и `.` и `,` → трактуем `.` как разделитель тысяч, `,` как десятичный
(RU-стиль). Нечисло → строка пропускается (`skipped`).

#### Исключение снятия наличных (`EXCLUDE_COMMENT_RE`)
Снятие наличных / обнал — это **перевод между счетами, а не трата**. Такие
строки отбрасываются **до** попадания в тоталы и до отправки в LLM (счётчик
`excluded` в логе). Ловятся по комментарию regex-ом: `снятие наличных`,
`снятие денег`, `обнал`, `mobilni keš/kes`, `smart atm`, `банкомат`.

### 3.4 Ручные доначисления (`manual_adjustments.csv`)
Если файл есть — строки **подмешиваются** к основным данным, выравниваясь по
именам полей (порядок колонок неважен). Используется для забытых регулярных
трат (например, аренда за прошлые месяцы — 7 строк по 850 €). Файл gitignored.
⚠️ **Риск двойного счёта**: если ту же аренду внести и в таблицу, и сюда — задвоится.
Защиты по дедупу пока нет (только предупреждение в доке).

### 3.5 Валюты (`fetch_rates`, `to_base`)
- Символы → ISO: `₽→RUB, дин→RSD, €→EUR, ¥→JPY, $→USD` (зеркалит `bot.py`).
- Курсы — live с `https://api.exchangerate-api.com/v4/latest/<BASE>` (без ключа).
  Формат: «сколько единиц валюты за 1 базовую». Перевод: `amount / rate`.
- Базовая валюта по умолчанию — **RSD** (динар). Поле `amount_base` = сумма в базовой.
- ⚠️ **Осознанное упрощение**: используется *сегодняшний* курс для всей истории.
  Пользователь ок с этим (евро↔динар почти стабилен, погрешность мала).

### 3.6 Нормализация
- **Спендеры** (`SPENDER_NORMALIZE`): `Azat→Азат`, а `Лиза/snezhi/снежи/лиса→Лиза`
  (это один человек: «Лиза» до августа, «snezhi» — после; подтверждено пользователем).
- **Категория-мусор**: если в `category` число (например `9000`) или пусто →
  `Bills`, если в комментарии «коммунал», иначе `Без категории`.
- **Аренда**: если в комментарии есть `аренд` → категория принудительно `Rent`
  (в таблице была ошибка ввода — аренда попадала в `Bills`).

### 3.7 Фильтр периода (`--from`, по умолчанию `2025-10`)
Скрывает месяцы раньше указанного (`YYYY-MM`). Ранние данные были «до того, как
семья осела», их временно прячем. `--from ""` показывает всё.

### 3.8 Группировка комментариев в темы
Два режима:
- **rules** (по умолчанию): `THEME_RULES` — упорядоченный список
  `(тема, [ключевые слова])`, матч по подстроке, **первое совпадение выигрывает**.
  Это семантическая группировка свободного текста БЕЗ внешнего LLM.
- **llm** (`--llm`): классификация комментариев в темы через внешнюю модель.
  Провайдер выбирается флагом `--provider`:
  - `anthropic` (**по умолчанию**, Claude): нужен `ANTHROPIC_API_KEY`,
    модель по умолчанию `claude-sonnet-5` (сильнее рассуждает — точнее
    классифицирует комментарии). SDK `anthropic`, вызов `messages.create`,
    ответ — строгий JSON `{комментарий: тема}` (парсер терпит ```json-обёртку).
  - `openai`: нужен `OPENAI_API_KEY`, модель по умолчанию `gpt-4o-mini`.

  Общий промпт для обоих провайдеров строит `build_grouping_prompt()`;
  диспетчеризация через таблицу `LLM_PROVIDERS` (провайдер → `(env-переменная,
  функция батча)`). Комментарии батчатся по 60, результат кэшируется **отдельно
  по провайдеру** в `output/llm_cache_<provider>.json` (`cache_path()`),
  повторные запуски шлют в модель только новые комментарии. Если ключа нет →
  «Без темы»; ошибка батча деградирует в «Прочее» (сборка не падает). Модель
  переопределяется `--model`.

  #### Гибрид: LLM + правила-fallback для «Прочее»
  LLM силён в смысле («капучино» → Кофе), но не знает **локальных названий**
  заведений/брендов («Bobo», «Sloj», «Осам»). Поэтому в `build()` после ответа
  Claude всё, что помечено **«Прочее»**, прогоняется через `classify_theme()`
  (правила знают эти названия). «Прочее» остаётся только у реально
  неклассифицируемого. На текущих данных это уронило «Прочее» с 34 до 8
  уникальных комментариев (в основном непрозрачные названия + снятие наличных).
  Логируется как «Rules fallback re-classified N row(s)».

  #### ⚠️ Закрытый список тем (обязательно для LLM)
  Блок «Оптимизация» в `template.html` раскидывает траты по тирам **по точному
  названию темы** (`THEME_TIER`). Поэтому LLM НЕ должен сочинять свободные метки:
  промпт даёт **закрытый список** канонических тем (`CANONICAL_THEMES` = ключи
  `THEME_RULES`) + «Прочее», и модель обязана выбрать одну из них. Дополнительно
  ответ прогоняется через `_snap_to_canonical()` (регистронезависимое совпадение
  с каноном, иначе «Прочее») — гарантия, что метки совпадут с `THEME_TIER`.
  Если добавляешь тему в `THEME_RULES` — она автоматически попадает и в список
  для LLM, и в тиры (не забудь прописать её в `THEME_TIER` в `template.html`).

Не подошедшее ни под одно правило → **«Прочее»**. Правила итеративно расширяли,
чтобы «Прочее» схлопнуть до реально нерасклассифицируемого.

#### ⚠️ Хрупкость классификатора и защита
Матч по подстроке + «первое совпадение» уже ломался трижды:
- `Фитпасс` ловился как «Метро» из-за `пасс` → убрали `пасс/pass`.
- `Чайник` мог уйти в «Продукты» из-за `чай` → «Дом и быт» поставили ВЫШЕ «Продуктов».
- `DM` перенесли из «Продуктов» в «Дом и быт» (по просьбе).
- `pok` сузили до `pokemon/pokémon/покемон` (иначе ловил бы «Покупку»).

Чтобы такие регрессии падали сразу, есть **самотест** `CLASSIFY_SELFTEST` +
`_run_classify_selftest()` (вызывается в начале `main()`): словарь
`комментарий → ожидаемая тема`, при расхождении — `AssertionError` на старте.
**Добавляешь ключевое слово → проверь, что самотест проходит.**

## 4. Клиентская логика (`template.html`)

Весь расчёт — на клиенте из вшитого `DATA`. Тёмная тема, Chart.js с CDN.

### 4.1 Глобальное состояние
- `mode` — валютный режим: `base` (всё в RSD по курсу) | `native` (исходные суммы).
- `monitorDim` — измерение монитора: `category | theme | who`.
- `excluded` — `Set` исключённых из анализа категорий.

### 4.2 Фильтрация
- `filteredRows()` — применяет фильтры (месяц/кто/категория/валюта) **и** исключения.
- `scopedRows()` — применяет **только** исключения категорий. Используется в
  помесячных расчётах (KPI «текущий месяц», MoM, инсайты «вырос/упал»), которые
  намеренно НЕ зависят от выбранного в фильтре месяца, но должны учитывать
  «убрать аренду» и т.п.
- `val(r)` — значение строки в текущем режиме: `amount_base` (base) или `amount` (native).
  ⚠️ В native агрегаты осмысленны только при выбранной одной валюте.

### 4.3 Исключение категорий (чипы «Исключить из анализа»)
Клик по чипу добавляет/убирает категорию в `excluded`; перечёркнутый = исключена.
Влияет на ВЕСЬ дашборд (KPI, инсайты, монитор, оптимизация, графики, таблица).
Сценарий: убрать `Rent`, чтобы смотреть на сокращаемые расходы.

### 4.4 KPI (`renderKpis`)
- **Всего** = `Σ val(r)` по отфильтрованным.
- **Средний чек** = Всего ÷ число операций (= средняя ОДНА трата, не за месяц/день).
- **Текущий месяц** = `Σ amount_base` за последний месяц в данных; стрелка ▲/▼ =
  изменение к предыдущему месяцу: `(тек − пред) / пред`. Не зависит от фильтра месяца.
- **Топ-категория** = категория с макс. суммой под фильтрами.

### 4.5 Инсайты (`renderInsights`) — фокус «мониторинг + где резать»
1. **Главная статья** — топ-категория и её доля (%).
2. **📈 Вырос сильнее всего** / **📉 сильнее всего упал** — макс. прирост/падение
   суммы категории (в RSD) за последний месяц к предыдущему, с % к базе. Считается
   по `scopedRows()` (независимо от фильтра месяца).
3. **🔁 Регулярные траты** — категории, встречающиеся минимум в
   `ceil(0.8 × число_месяцев)` месяцах (но не меньше 2). Это «фиксированный»
   регулярный расход; показываем среднее/мес = `total_base / число_месяцев`.
4. **✂️ Кандидат на сокращение** — крупнейшая регулярная **необязательная** (`flex`)
   категория по среднему/мес (аренда/счета исключены через тиры).

### 4.6 📅 Монитор «Расходы по месяцам» (`renderMonthlyMonitor`) — ГЛАВНЫЙ блок
Теплокарта-матрица: **строки = статьи, столбцы = месяцы**, ячейка = сумма за месяц
в текущем режиме (`val`). Чем ярче (насыщеннее синий), тем больше потрачено —
прозрачность `0.10 + 0.80·(v/maxCell)`. Показываем топ-18 статей по итогу.
- Колонка **Σ** — итог за период; **тренд** — последний месяц к предыдущему (▲/▼ %).
- Строка **Итого** — сумма по месяцам.
- Наведение на ячейку → разбивка по валютам (data-bd).
- Переключатель **Категории / Темы / Кто** (`monitorDim`) — так «кто тратит»
  стал режимом группировки, а не отдельной цифрой.

### 4.7 💡 Оптимизация «На чём можно сэкономить» (`renderOptimization`)
Отвечает на «что оптимизировать, а что можно НЕ оптимизировать».
Категории разбиты на 3 тира (`TIER_OF` / `TIERS`):

| Тир | Эмодзи | Категории | Смысл |
|---|---|---|---|
| `fixed` | 🔒 Обязательные | Rent, Bills | фундамент, почти не режется |
| `base` | ⚙️ Базовые | Food, Transport, Household, Health and Beauty | нужное, можно ужать |
| `flex` | 💸 Необязательные | Delivery food, Eating out, Sweets and snacks, Entertainment, Clothes, Sport, Travels, Technics, Other | главный потенциал экономии |

**Тир определяется построчно (`rowTier`), глубже категории:**
сначала по **теме** комментария (`THEME_TIER` — точный сигнал), и только если темы
нет (нет комментария / «Прочее») — откат на категорию (`TIER_OF`). Это разделяет,
например, `Такси` (💸) и `Метро` (⚙️) внутри одной категории `Transport`, или
доставку/кофе, ошибочно записанные в `Food`.

`THEME_TIER` (тема → тир), согласовано с пользователем:
- 🔒 fixed: `Жильё и коммуналка`, `Документы и бюрократия` (ВНЖ/виза/нотариус — неизбежно);
- ⚙️ base: `Супермаркет и продукты`, `Метро и транспорт`, `Здоровье и аптека`, `Дом и быт`;
- 💸 flex: всё остальное (Такси, Кофе, Доставка, Кафе, Красота, Спорт и танцы, Одежда,
  Развлечения, Алкоголь, Сладости, Шопинг, Техника, Путешествия, Подарки, Суши, Фастфуд,
  Образование — курсы/репетиторы, необязательное по просьбе пользователя).

**`Подписки и сервисы` дробится по комментарию** (`rowTier` + `CONNECTIVITY_RE`):
интернет/связь/SIM (`internet|интернет|sim|airalo|yettel|esim|роуминг|связь|мобильн`)
→ ⚙️ base; всё прочее (ChatGPT, Spotify, игры…) → 💸 flex.

**Бакеты в панели** (`bucketLabel`): обычно = категория, но если категория реально
смешивает тиры — дробится на под-бакеты **`Категория · тир`** (например
`Transport · 💸`, `Food · ⚙️`). На текущих данных дробятся: Transport, Food,
Eating out, Delivery food, Other, Health and Beauty, Bills.

Суммы тиров (`tierSum`) считаются **построчно и точно** (без округления бакетов).
`Health and Beauty` больше не особый случай — он покрывается общим механизмом
(`Красота`→flex, `Здоровье и аптека`→base). Всё это применяется **только** в лупе
оптимизации (панель + инсайты «регулярные»/«кандидат»). Монитор, таблица и прочие
графики работают с сырой категорией. Drill-down с бара оптимизации фильтрует строки
через `bucketLabel(r, lastCatTiers)`.

Блок показывает:
- **3 карточки тиров**: среднее/мес (`tierSum / число_месяцев`) и доля бюджета (%).
- **Зелёный вывод (takeaway)**: сумма необязательного/мес, потенциальная экономия
  при −30% (`flexMonth × 0.3`), самая весомая необязательная статья, и
  выросли/упали необязательные в последнем месяце.
- **Горизонтальный бар** всех категорий по среднему/мес, **покрашенный по тиру**
  (серый/жёлтый/красный) — мгновенно видно, где «красные» большие = где резать.

Все суммы здесь — в **базовой валюте** (бюджетирование нагляднее в одной валюте),
независимо от валютного режима.

### 4.8 Прочие графики (`renderCharts`)
По категориям (donut), динамика по месяцам (line), кто тратит (bar), по валютам
(donut, всегда исходные суммы), темы трат (bar, топ-15), средний чек по категориям
(`сумма / число операций` категории, топ-12), траты по дням недели (выходные —
оранжевым), состав по месяцам (stacked, топ-7 категорий).

### 4.9 Explorer «Как сгруппированы комментарии» (`renderGrouping`)
Раскрываемый список тем: сумма, число операций, ключевые слова правила и сами
комментарии, попавшие в тему. Видно, что именно в «Прочем».

### 4.10 Таблица операций (`renderTable`)
Сортировка по клику на заголовок; даты сортируются хронологически (парсятся в `Date`,
а не как строки). В режиме base наведение на сумму показывает исходную сумму.

### 4.11 Drill-down «что внутри» (`openDrill`)
Клик по сектору «По категориям», бару «Оптимизации» или по названию строки
монитора открывает модалку с раскладкой: **на что именно** потрачено внутри
группы. Группировка внутри — по сырому тексту комментария (это и есть
«Массаж/Маникюр/…»), отсортировано по сумме, с тегами тем, числом операций и
долей. `openDrill(dim, value)` универсален: `dim` = `category|theme|who`
(для монитора берётся текущий `monitorDim`, для графиков — `category`).
Закрытие: крестик, клик по фону, Esc.

### 4.12 Тултипы
Плавающий `#tip`:
- `data-bd` — разбивка по валютам (как есть) для агрегатов.
- `data-tip` — пояснение «как считается» у KPI и заголовков (иконка `i`).

### 5.1 Авто-обновление (`run.sh`, `schedule.sh`) и доставка в Telegram

**Архитектура доставки:** дашборд собирается **только на build-машине** (Маке).
`run.sh` всегда идёт с `--publish`: после сборки скрипт грузит HTML в Telegram
(`sendDocument` в чат `DASHBOARD_CHAT_ID` через `BOT_TOKEN`) и пишет полученный
`file_id` + timestamp в лист `Meta` таблицы (`Meta!B1`). Telegram-бот **не
собирает** дашборд — по `/dashboard` он читает `Meta!B1` и пере-отправляет тот же
файл по `file_id` (мгновенно, с любого устройства; `file_id` переиспользуется
одним и тем же ботом в любом чате). Функции: `publish_to_telegram()` +
`_store_file_id_in_sheet()` в `build_dashboard.py`; чтение на стороне бота —
`get_dashboard_file_id()` в `bot.py`.

**По требованию:** `./run.sh [доп. аргументы build_dashboard.py]`. Скрипт при
первом запуске создаёт изолированный `venv/` и ставит `requirements.txt`, затем
собирает дашборд из **Google Sheets** с классификацией через **Claude**
(`--llm --provider anthropic --publish`). Лог — `output/build.log`. Собрать без
отправки: `python build_dashboard.py --llm` (без `--publish`).

**Раз в месяц:** `./schedule.sh install` кладёт в `~/Library/LaunchAgents/`
plist `com.zovminga.familybot.dashboard.plist` (генерится с абсолютным путём к
`run.sh`, в репозиторий не коммитится) и грузит его через `launchctl`.
`StartCalendarInterval` = 1-е число, 09:00 (меняется `RUN_DAY`/`RUN_HOUR`/
`RUN_MINUTE`). Выбран **launchd**, а не cron: при спящем/выключенном Mac launchd
догоняет пропущенный запуск при пробуждении. Подкоманды: `install` / `uninstall`
/ `status` / `run` (разовый прогон через launchd). Лог — `output/launchd.log`.

⚠️ Для авто-обновления из Sheets нужен `family_bot/secrets/creds.json`
(service account) и таблица, расшаренная на его email; и `ANTHROPIC_API_KEY`
в `family_bot/.env`.

## 6. Ключевые решения и оговорки

- **Курс — сегодняшний на всю историю** (упрощение, принято пользователем).
- **`Лиза = snezhi`** — один человек.
- **Аренда форсится в `Rent`** по слову `аренд` (фикс ошибки ввода).
- **Данные < 2025-10 скрыты** по умолчанию (можно вернуть `--from ""`).
- **Тиры оптимизации** — субъективная классификация; правится в одном месте
  (`TIER_OF` в `template.html`). Спорные: `Sport` (отнесён к flex),
  `Health and Beauty` (base, хотя «Beauty» оптимизируема), `Other` (flex).
- **Классификатор тем** хрупок к новым ключам — всегда гоняй самотест.
- **Ручные доначисления** могут задвоиться, если те же траты появятся в таблице.
